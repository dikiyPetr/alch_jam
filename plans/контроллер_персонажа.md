# План: Контроллер персонажа-слизня (прототип)

## Цель

Прототип для тестирования механики деления и слияния слизня в top-down 3D сцене.
Персонаж отображается через спрайт/билборд. Все значения — заглушки, удобные для итерации.

---

## Модель данных: пул и распределение HP

### Ключевые правила

1. **Пул глобален** — все особи (большие и малые) всегда входят в единый `SlimePool`. Пул хранит суммарный `totalMaxHp` всей популяции.
2. **Количество особей выводится через HP** — `unitCount = totalMaxHp / hpPerUnit`, где `hpPerUnit` — константа (например, 100). Явный список особей не нужен.
3. **Большая особь** (`BigSlime`) владеет своей долей `maxHp` из пула. Её характеристики (размер, скорость, урон) масштабируются пропорционально этой доле.
4. **Малая особь** (`SmallSlime`) владеет ровно `hpPerUnit` из доли той `BigSlime`, которая её выпустила.
5. **Несколько больших особей** — пул делится поровну между всеми активными `BigSlime`. Каждая `BigSlime` из своей доли может дополнительно выделять `SmallSlime`.
6. **Возврат в пул** — если малая особь уничтожена или потеряна (не вернулась к `BigSlime`), её `hpPerUnit` возвращается в `pool.availableMaxHp` для перераспределения.

### Схема распределения HP

```
SlimePool.totalMaxHp = 1000
  ├─ BigSlime A  → доля 500 (50%)
  │    ├─ внутри тела:  400 hp (4 единицы)
  │    └─ SmallSlime x2: 100 hp (2 × 50 hpPerUnit*)
  └─ BigSlime B  → доля 500 (50%)
       ├─ внутри тела:  450 hp (4.5 единицы → округляется вниз = 4)
       └─ SmallSlime x1: 50 hp (1 × 50 hpPerUnit)

* hpPerUnit может отличаться от доли BigSlime — настраивается отдельно
```

## Архитектура

### Классы

```
SlimePool           — глобальный пул: totalMaxHp, распределение долей между BigSlime
BigSlime            — большая особь: своя доля из пула, движение, масштаб, Split/Merge
SmallSlime          — малая особь: hpPerUnit из доли своей BigSlime, следование/слияние
SlimeController     — ввод игрока → команды первой (или выбранной) BigSlime
SlimeBillboard      — поворот спрайта к камере
SlimeDebugUI        — отображение состояния пула и всех BigSlime в рантайме
```

## Детальный план реализации

### 1. SlimePool (singleton MonoBehaviour или ScriptableObject)

**Поля:**
- `float totalMaxHp` — суммарный максимальный HP всей популяции (постоянен пока особи не гибнут)
- `float hpPerUnit = 50f` — HP одной единицы слизня (константа, настраивается в Inspector)
- `List<BigSlime> bigSlimes` — зарегистрированные большие особи

**Вычисляемые свойства:**
- `int TotalUnitCount` → `Mathf.FloorToInt(totalMaxHp / hpPerUnit)`
- `float SharePerBig` → `totalMaxHp / bigSlimes.Count` — доля каждой большой особи

**Методы:**
- `RegisterBig(BigSlime b)` — добавляет BigSlime, перераспределяет доли
- `UnregisterBig(BigSlime b)` — убирает BigSlime, возвращает её долю в `totalMaxHp`, перераспределяет
- `ReturnHp(float amount)` — малая особь потеряна: прибавляет `amount` к `totalMaxHp`, инициирует перераспределение
- `RedistributeShares()` — всем BigSlime устанавливает `assignedMaxHp = SharePerBig` (текущее HP при этом не изменяется, только максимум)

---

### 2. BigSlime : MonoBehaviour

**Поля:**
- `SlimePool pool`
- `float assignedMaxHp` — доля этой BigSlime из пула (устанавливается через `pool.RedistributeShares`)
- `float containedMaxHp` — HP, остающийся внутри тела (= assignedMaxHp минус доли выпущенных SmallSlime)
- `float currentHp` — текущее HP (≤ containedMaxHp); при получении урона уменьшается
- `List<SmallSlime> spawnedSmalls` — выпущенные малые особи
- `float minScale, maxScale` — пределы масштабирования тела
- `float splitRadius = 1.5f`
- `float mergeRadius = 0.5f`
- `GameObject smallSlimePrefab`

**Вычисляемые свойства:**
- `int ContainedUnitCount` → `Mathf.FloorToInt(containedMaxHp / pool.hpPerUnit)`
- `float SizeRatio` → `containedMaxHp / assignedMaxHp` — относительный размер тела
- `float MoveSpeed` → базовая скорость × `SizeRatio` (больше слизней внутри = медленнее? или быстрее — настраивается)

**Движение:**
- `CharacterController` или `Rigidbody`, движение по XZ
- Скорость берётся из `MoveSpeed`

**Масштаб тела:**
- `Update`: `transform.localScale = Vector3.one * Mathf.Lerp(minScale, maxScale, SizeRatio)`

**Split(int count):**
1. Проверить: `ContainedUnitCount > count` (иначе нельзя — в теле должна остаться хотя бы 1 единица)
2. `float hpToSplit = count * pool.hpPerUnit`
3. `containedMaxHp -= hpToSplit`
4. Создать `count` SmallSlime, каждому передать `hpPerUnit` и ссылку на `this`
5. Добавить в `spawnedSmalls`

**Merge():**
- Всем SmallSlime в `spawnedSmalls` установить режим `Merging`
- При достижении `mergeRadius`: `containedMaxHp += small.hpPerUnit`, уничтожить SmallSlime

**OnSmallLost(SmallSlime small)** — вызывается когда SmallSlime уничтожена без слияния:
- `spawnedSmalls.Remove(small)`
- `pool.ReturnHp(small.hpPerUnit)` — HP уходит обратно в общий пул

**TakeDamage(float amount):**
- Уменьшает `currentHp`
- Если `currentHp <= 0`: `pool.UnregisterBig(this)`, уничтожить GameObject

---

### 3. SmallSlime : MonoBehaviour

**Поля:**
- `float hpPerUnit` — берётся из `pool.hpPerUnit` при создании
- `float currentHp`
- `BigSlime owner` — BigSlime, выпустившая эту особь
- `int indexInChain` — позиция в цепочке
- `float followOffset = 0.6f`
- `SlimePool pool`

**Состояния (`enum SlimeState`):**
- `Following` — следует за owner (или предыдущим в цепочке)
- `Merging` — движется к owner для слияния

**Update:**
- `Following`: цель = `owner.transform.position - chain_offset * indexInChain`; `MoveTowards` со скоростью `pool.hpPerUnit`-пропорциональной (или фиксированной для прототипа)
- `Merging`: движется к `owner.transform.position`; при достижении `mergeRadius` вызывает `owner.Merge(this)`

**TakeDamage / смерть:**
- `currentHp -= amount`
- Если `currentHp <= 0`: `owner.OnSmallLost(this)`, уничтожить GameObject

---

### 4. SlimeController : MonoBehaviour

Обрабатывает ввод, передаёт команды. В прототипе управляет первой `BigSlime` из `pool.bigSlimes`.

**Ввод (New Input System):**
- `Move` (WASD) → `targetBig.SetInput(dir)`
- `Split` (Q) → `targetBig.Split(1)`
- `Merge` (E) → `targetBig.Merge()`

---

### 5. SlimeBillboard : MonoBehaviour

```csharp
void LateUpdate() {
    transform.rotation = Camera.main.transform.rotation;
}
```

---

### 6. SlimeDebugUI

`OnGUI`-оверлей:
- `pool.totalMaxHp` / `pool.TotalUnitCount`
- Для каждой BigSlime: `assignedMaxHp`, `containedMaxHp`, `ContainedUnitCount`, `spawnedSmalls.Count`
- `pool.SharePerBig`

## Сцена прототипа

**Иерархия:**
```
[Scene]
  GameManager       ← SlimePool (singleton), SlimeController
  BigSlime_A        ← BigSlime, CharacterController, SlimeBillboard
    Sprite          ← SpriteRenderer (квадрат-заглушка)
  SmallSlimes       ← пустой контейнер для SmallSlime
  Ground            ← Plane
  Camera            ← top-down (~60°), следует за BigSlime_A
  DebugCanvas       ← SlimeDebugUI
```

---

## Порядок реализации

1. `SlimePool` — глобальный пул, перераспределение долей
2. `SlimeBillboard` — сразу видно спрайт
3. `BigSlime` — регистрация в пуле, движение, масштаб по `SizeRatio`
4. `SlimeController` — ввод
5. `SmallSlime` — следование и слияние
6. `BigSlime.Split / Merge / OnSmallLost` — полный цикл деления
7. Тест нескольких BigSlime — создать вторую BigSlime вручную, проверить перераспределение
8. `SlimeDebugUI` — наблюдение за состоянием пула

---

## Что НЕ входит в прототип

- Анимации, звуки
- Враги и боевая система
- Прокачка / скилы / модификаторы
- Карта в стиле Slay the Spire

## Открытые вопросы для тестирования

- Перераспределение долей между BigSlime происходит мгновенно или плавно (lerp)?
- Что происходит с `currentHp` BigSlime при перераспределении (если доля выросла — растёт ли текущее HP)?
- Как выглядит цепочка малых особей — строго линейно или роем?
- Минимальное количество единиц, остающихся в теле BigSlime после Split (1 или больше)?
- Урон по большой особи уменьшает только `currentHp` или также `containedMaxHp` (т.е. "убивает" единицы внутри)?
